# Technical Architecture вЂ” gaal

## Overview

`gaal` is a Go (1.22) CLI tool that manages three types of local resources from a single YAML configuration file:

| Resource | Description |
|----------|-------------|
| **Repositories** | Multi-protocol repositories (git, hg, svn, bzr, tar, zip) |
| **Skills** | `SKILL.md` file collections installed into AI agent directories |
| **MCPs** | MCP (*Model Context Protocol*) server configuration entries injected into JSON files |

## Execution Flow

```
main.go
  в””в”Ђв–є cmd.Execute() [Cobra]
        в””в”Ђв–є rootCmd.PersistentPreRunE        вЂ” runs before every sub-command
              в”њв”Ђв–є printBanner()             вЂ” ASCII-art banner (TTY only, skipped with --no-banner)
              в”њв”Ђв–є setupLogger()             вЂ” configure slog (level, log-file)
              в””в”Ђв–є applyOptions()            вЂ” activate sandbox if --sandbox set
                    в””в”Ђв–є engineOpts (package-level var shared by all commands)
        в”њв”Ђв–є (no sub-command given)          вЂ” banner + help listing
        в””в”Ђв–є cmd.runSync() / cmd.runStatus()
              в”њв”Ђв–є config.LoadChain()         вЂ” merge global в†’ user в†’ workspace
              в”њв”Ђв–є engine.NewWithOptions()    вЂ” instantiate the 3 managers
              в””в”Ђв–є engine.RunOnce()           вЂ” trigger synchronisation
                    в”њв”Ђв–є repo.Manager.Sync()   (parallel goroutines)
                    в”њв”Ђв–є skill.Manager.Sync()  (sequential per source)
                    в””в”Ђв–є mcp.Manager.Sync()    (sequential per entry)
```

In `--service` mode (`gaal sync --service`), `engine.RunService()` wraps `RunOnce()` in a `time.Ticker` loop with clean `SIGTERM` handling via `context.Context`.

## Package `internal/engine`

This is the **single coupling point** between the configuration and the three managers. It is instantiated by CLI commands and contains no business logic of its own вЂ” it only sequences the calls.

### `Options.WorkDir` parameter

During a normal run, `workDir` defaults to the current working directory (`os.Getwd()`). In sandbox mode (`--sandbox`), it is redirected to an isolated subdirectory, preventing any writes outside the sandbox.

### Initialisation вЂ” `PersistentPreRunE`

`printBanner()`, `setupLogger()` and `applyOptions()` are called **once** in `rootCmd.PersistentPreRunE` before any sub-command runs. The resulting `engine.Options` is stored in the package-level `engineOpts` variable and reused by `runSync` and `runStatus`. Adding a new sub-command requires no boilerplate вЂ” banner, sandbox and logging are automatically active.

### Banner (`cmd/banner.go`)

`printBanner()` writes a cyan ASCII-art header to stdout. It is a **no-op** when stdout is not a TTY (pipes, redirections, CI). It can be suppressed at runtime with the global flag `--no-banner`.

## Package `internal/config`

### Configuration hierarchy

`gaal` loads and merges up to three configuration files in priority order (lowest to highest):

| Priority | Level | Linux | macOS | Windows |
|----------|-------|-------|-------|---------|
| 1 (lowest) | **Global** | `/etc/gaal/config.yaml` | `/etc/gaal/config.yaml` | `%PROGRAMDATA%\gaal\config.yaml` |
| 2 | **User** | `$XDG_CONFIG_HOME/gaal/config.yaml` (default: `~/.config/gaal/config.yaml`) | `~/Library/Application Support/gaal/config.yaml` | `%AppData%\gaal\config.yaml` |
| 3 (highest) | **Workspace** | value of `--config` flag (default: `gaal.yaml` in CWD) | в†ђ same | в†ђ same |

Each level is optional. Missing files are silently skipped (a `DEBUG` log is emitted for each absent file). At least one file must exist.

The Global and User paths are resolved at runtime via Go standard library APIs:
- Global: hardcoded `/etc/gaal/config.yaml` on Linux/macOS; `os.Getenv("PROGRAMDATA")` on Windows.
- User: `os.UserConfigDir()` вЂ” respects `XDG_CONFIG_HOME`, `~/Library/Application Support`, and `%AppData%` on the respective OS.

### Merge strategy

`LoadChain(workspacePath)` loads each present file through the standard `Load()` pipeline (parse в†’ validate в†’ `expandPaths()`), then merges them bottom-up:

- **`repositories`** вЂ” merged by path key. Higher-priority files win on conflict: a workspace entry for `src/foo` replaces any global or user entry for the same key.
- **`skills` / `mcps`** вЂ” **accumulated**: entries from all levels are concatenated and all applied. This lets a global config install company-wide skills while a workspace config adds project-specific ones without overwriting.

### Path resolution

`expandPaths()` is called per file, anchored to that file's own directory. This means relative repository paths in `/etc/gaal/config.yaml` are resolved relative to `/etc/gaal/`, and relative paths in `gaal.yaml` are resolved relative to the project root.

### Main types

| Type | Role |
|------|------|
| `Config` | Root: `Repositories`, `Skills`, `MCPs` |
| `RepoConfig` | [vcstool](https://github.com/dirk-thomas/vcstool)-compatible repository entry |
| `SkillConfig` | Source + target agent list + scope (global/project) |
| `MCPConfig` | Name + JSON target + inline definition or remote URL |

## Package `internal/repo`

### `VCS` interface

All backends implement four methods:

- `Clone(ctx, url, path, version)` вЂ” initial clone
- `Update(ctx, path, version)` вЂ” update an already-cloned repository
- `IsCloned(path)` вЂ” checks for the metadata directory (`.git/`, `.svn/`, etc.)
- `CurrentVersion(ctx, path)` вЂ” human-readable version (tag, branch, commit, revision)

### Available backends

| Type | Class | Required binary | Local detection |
|------|-------|----------------|-----------------|
| `git` | `VcsGit` | **none** (go-git, pure Go) | presence of `.git/` |
| `hg` | `VcsMercurial` | `hg` | presence of `.hg/` |
| `svn` | `VcsSVN` | `svn` / `svnversion` | presence of `.svn/` |
| `bzr` | `VcsBazaar` | `bzr` | presence of `.bzr/` |
| `tar` / `zip` | `VcsArchive` | none (stdlib) | presence of the target directory |

All five types are enforced at compile time via `var _ VCS = (*VcsXxx)(nil)` assertions in `vcs.go`.

For subprocess backends (`VcsMercurial`, `VcsSVN`, `VcsBazaar`), `requireBinary(name)` is the **first call** in every method вЂ” it returns a clear error message rather than a cryptic exec failure.

### Parallelism

`repo.Manager.Sync()` spawns one goroutine per repository. Errors are collected via a buffered channel and aggregated after a `sync.WaitGroup`. This prevents a slow repository from blocking the others.

### Archives (`tar` / `zip`)

The `version` field in `RepoConfig` acts as a **strip prefix** during extraction, mirroring vcstool behaviour. There is no incremental update mechanism for archives вЂ” `Update()` is a no-op.

## Package `internal/skill`

### Skill source lifecycle

```
resolveSource()
  в”њв”Ђ local path в†’ returned as-is
  в””в”Ђ GitHub shorthand / URL в†’ git clone into <cache-dir>/gaal/skills/<cache-key>/
        в””в”Ђ already cloned в†’ git fetch + pull (silent update)

discoverSkills()   вЂ” recursive SKILL.md search in known subdirectories
filterSkills()     вЂ” filter by the `select:` list
resolveAgents()    вЂ” expand ["*"] to detected installed agents
installSkill()     вЂ” copy the skill directory to the agent's target directory
```

### Why a local skill cache?

Remote sources (GitHub or URLs) are **cloned locally** for two reasons:

1. **Idempotence**: a second `sync` call does not re-clone вЂ” it runs `git pull` instead.
2. **Offline resilience**: if the source is temporarily unreachable, the previously cached version remains available.

The cache root is resolved via `os.UserCacheDir()` (OS-appropriate, never hardcoded):

| OS | Cache root | Full skill cache path |
|----|-----------|----------------------|
| Linux | `$XDG_CACHE_HOME` or `~/.cache` | `~/.cache/gaal/skills/` |
| macOS | `~/Library/Caches` | `~/Library/Caches/gaal/skills/` |
| Windows | `%LocalAppData%` | `%LocalAppData%\gaal\skills\` |

Each source is stored under a key derived from its URL (`urlToCacheKey()`), guaranteeing no collision between two different sources.

In **sandbox** mode, `$HOME` is redirected, so `os.UserCacheDir()` falls back to the sandbox tree вЂ” nothing is written to the user's real directories.

### Installation scope: `global` vs project

| `global: false` (default) | `global: true` |
|---------------------------|----------------|
| `<project>/<agent-dir>/skills/` | `~/.<agent>/skills/` |
| Versionable alongside the project | Shared across all projects |

### Agent registry (`knownAgents`)

A static `map[string]AgentInfo` in `agents.go` registers ~20 known agents (Copilot, Claude, Cursor, Goose, etc.) with their project-relative and global `$HOME` skill directories. This registry powers the automatic resolution when `agents: ["*"]` is used.

Detection checks whether the agent's parent directory **already exists on disk**, avoiding the creation of directories for agents that are not installed.

## Package `internal/mcp`

### Merge strategy

Each MCP entry is **upserted** into the target JSON file under the `mcpServers.<name>` key. Existing keys not managed by gaal are preserved (round-trip). The operation is atomic: write to a temporary file + `os.Rename()`.

### Supported sources

| Mode | Description |
|------|-------------|
| `inline:` | Full definition in YAML (command, args, env) |
| `source:` | Remote URL returning a JSON document `{mcpServers: {...}}` |

## Package `internal/runner`

An abstraction layer over `os/exec.Command` that adapts its output behaviour to the current log level:

| Level | Behaviour |
|-------|-----------|
| `DEBUG` (`--verbose`) | subprocess stdout + stderr logged line-by-line via `slog.Debug` |
| `INFO` (default) | output captured silently + TTY spinner. On failure, captured output is emitted via `slog.Error`. |

---

## Package `internal/logger`

### Handlers

| Handler | Destination | Format |
|---------|-------------|--------|
| `ConsoleHandler` | `stderr` | Compact, colorized (ANSI), auto-detected TTY |
| `slog.JSONHandler` | file (`--log-file`) | JSON Lines with `host` and `time` fields |

When `--log-file` is provided, a `teeHandler` fans out to both handlers simultaneously.

### Spinner (`progress.go`)

Only one spinner is active at a time (mutex `ttyMu`). The spinner and `ConsoleHandler` share this mutex to prevent interleaved writes to `stderr`. The spinner is a no-op when `stderr` is not a TTY (CI environments, shell redirections).

## Sandbox Mode (`--sandbox <dir>`)

Sandbox mode is designed for testing and CI. Its implementation relies on **a single primitive**: `os.Setenv("HOME", sandboxDir)`, which causes all `os.UserHomeDir()`, `os.UserConfigDir()`, and `os.UserCacheDir()` calls to resolve inside the sandbox tree.

Cascading effects:
- `os.UserHomeDir()` в†’ `sandboxDir`
- Skill cache (`os.UserCacheDir()`) в†’ `sandboxDir/.cache/gaal/skills/` (Linux)
- All `global: true` skill paths в†’ `sandboxDir/.<agent>/skills/`
- All MCP targets (`~/.config/...`) в†’ `sandboxDir/.config/...`
- `WorkDir` в†’ `sandboxDir/workspace/`

No real user resources are modified.

## Build, Tests and Version Injection

The binary is produced in `dist/gaal` (git-ignored). The version and build timestamp are injected at compile time via `ldflags`:

```
-X gaal/cmd.Version=$(git describe --tags --always --dirty)
-X gaal/cmd.BuildTime=$(date -u +%Y-%m-%dT%H:%M:%SZ)
```

Without these flags (e.g. `go run .`), values fall back to `"dev"` / `"unknown"`.

### Makefile targets

| Target | Description |
|--------|-------------|
| `make build` | Compile the binary to `dist/gaal` |
| `make test` | Run the full test suite (`go test ./...`) |
| `make coverage` | Run tests with coverage and generate four reports in `report/` (git-ignored) |
| `make lint` | Run `go vet` |
| `make sandbox` | One-shot sync in an isolated `/tmp` directory |
| `make clean` | Remove `dist/` |

### Coverage reports (`make coverage`)

All outputs are written to `report/` (git-ignored):

| File | Tool | Description |
|------|------|-------------|
| `coverage.html` | `go tool cover` | Standard Go source heat-map |
| `coverage-golang.html` | `gocov-html -t golang` | Tabular report, golang theme |
| `coverage-kit.html` | `gocov-html -t kit` | Tabular report, AdminKit dark theme |
| `coverage-treemap.svg` | `go-cover-treemap` | SVG heatmap by package |
